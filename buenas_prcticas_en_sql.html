BUENAS PRÁCTICAS EN SQL

Fuente. https://es.linkedin.com/pulse/2-de-las-mejores-pr%C3%A1cticas-para-eliminar-delete-o-en-ocampo-ospina

 La siguiente lista se constituye en un compendio de buenas prácticas a implementar en bases de datos relacionales como Oracle.
  1. Establecer convenciones en el nombrado de la base de datos y sus objetos, estandarizarlo y apegarse a él.
  2. La normalización debe ser usada cuando sea requerida para optimizar el performance. Una baja normalización puede repercutir en una repetición de datos, una sobre normalización puede tener efectos en el rendimiento a causa de las excesivas uniones entre tablas para extraer datos. Se debe mantener un equilibro.
  3. Normalizar, mínimo, a 3NF.
  4. Imágenes y columnas de tipo blob no deben estar definidas en tabla frecuentemente requeridas para evitar problemas de rendimiento. Estos datos deben ser puestos en tablas separadas relacionadas por un identificador.
  5. Crear índices sobre los campos de las tablas, teniendo en cuenta las consultas más populares de los usuarios.
  6. Definir tipos de datos convenientes, según el dominio de cada atributo.
  7. Evitar tipos de datos BLOB, CLOB, XML que, aunque están implementados en los motores, no son eficientes. Es mejor manejar la ruta de acceso de donde están estos tipos de objetos.
  8. En lo posible evitar usar SQL Dinámico ya que facilita el SQL Injection.
  9. Minimizar el uso de valores nulos.
  10. Documentar procedimientos almacenados, triggers, funciones y demás programas. Además, documentar el modelo conceptual, lógico y físico de la base de datos.
  11. Evitar usar SELECT * en las consultas. Poner los campos específicos a consultar.
  12. Mantén los passwords encriptados por seguridad. Desencríptalos en la aplicación si es necesario.
  13. Usa constraints (llaves foraneas, Checks, valores no nulos, etc) para la integridad de datos. No hagas todo el control desde el código de la aplicación.
  14. En lo posible, evitar el uso de cursores. Internamente genera muchos árboles que consumen mucho recurso.
  15. Evitar usar, con el LIKE, comodines al principio de la palabra, por ejemplo, LIKE ‘%con’.
  16. Evitar hacer consultas por operadores de desigualdad, <>, not.
  17. Chequear, periódicamente, el plan de ejecución de las consultas.
  18. Usar SET NOCOUNT OFF / FEEDBACK OFF para evitar trafico innecesario en la red.
  19. No permitir a las aplicaciones FRONT END consultar o manipular datos directamente usando sentencias SQL.
  20. Accesar a las tablas siempre en el mismo orden, en los procedimientos, triggers y/o funciones. Esto, para evitar posibilidades de deadlocks.
  21. Evitar transacciones muy largas ya que reducen la concurrencia de la base de datos.
  22. Realizar manipulaciones de cadenas, concatenaciones, numerado de registros, conversiones en el FRONT END.
  23. Siempre que vaya a hacer un JOIN entre dos tablas, ponga de primero la tabla que tenga menor número de registros.
  24. Prefiere los JOINS en todas sus formas en el FROM, antes que en el WHERE.
  25. Nunca usar sentencias LIKE al principio del WHERE.
  26. Usar el ORDER BY solo si es estrictamente necesario.
  27. Si una tabla va a contener pocos registros, tratar de no crearle índices.
  28. Un servidor de base de datos y un servidor web deben estar en máquinas diferentes. Esto provee de mas seguridad y separan la carga de trabajo en dos CPUs y memoria diferentes.
  29. Utilice vistas materializables para consultas recurrentes.
  30. Definir políticas de backups.
  31. En lo posible, usar char, en vez de varchar.
  32. En los INSERTS, especificar los campos.
  33. Para valores booleanos, usar el tipo de dato bit.
  34. No darle rol de administrador a cualquier usuario.

Es de aclarar que las anteriores son solo sugerencias que se deben tener en cuenta, evaluando el contexto donde se van a aplicar, es decir, dependiendo de diversos factores, es posible no acatar la buena práctica.
